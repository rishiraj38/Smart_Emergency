# 🚨 Smart City Emergency Dispatch — Developer Guide

> **Read this first.** This document explains the entire project from scratch, what every endpoint does, why you see results "automatically", and how to actually use the system.

---

## 1. What Is This Project?

This is a **simulated 911 dispatch center** built as a Reinforcement Learning (RL) environment.

Think of it like a video game:
- 📞 Fake 911 calls come in (car accidents, fires, medical emergencies)
- 🧠 An AI agent (or you, via API) must decide what to do with each call
- 🚑 You classify calls, dispatch ambulances/fire trucks/police to events
- ⏱️ Events have urgency — if you wait too long on a CRITICAL event, people "die" and the episode ends
- 🏆 You get a score based on how many events you resolved

**The goal:** Train an LLM to play this "game" better than a simple rule-based bot.

---

## 2. Why Do I See Results Without "Sending" Anything?

This confuses most developers at first. Here's the key insight:

> **The environment generates the world. You don't send data — you send DECISIONS.**

### Normal API (what you're used to):
```
POST /users  →  You send: {"name": "John"}  →  Server saves it  →  Returns the user
```
You provide the data. The server stores it.

### This RL Environment API (what this project does):
```
POST /reset  →  You send: {"task_id": 1}  →  Server GENERATES a whole city with calls  →  Returns the first observation
POST /step   →  You send: {"command": "CLASSIFY", "severity": "CRITICAL"}  →  Server SIMULATES what happens  →  Returns the new state
```

You provide **decisions**. The server simulates an entire world.

### Analogy: Chess
Imagine a chess API:
- `POST /reset` → Server sets up the board, returns the board state
- `POST /step {"move": "e2e4"}` → Server moves your piece, the opponent responds, returns the new board
- You didn't "send" the board. The server **is** the board.

**That's exactly what this project does, but with emergency dispatch instead of chess.**

## 3. The Complete Flow (Step by Step)

Here is the **exact** sequence of what happens when you use this API:

### Step 1: Reset — Start a New Episode

```http
POST /reset
Body: {"task_id": 1}
```

**What happens inside the server:**
1. Builds a city graph (20 nodes connected by roads)
2. Places emergency vehicles at stations (ambulances, fire trucks, police)
3. Generates 15 fake 911 calls (for Task 1) with random events
4. Picks the first call from the queue
5. Returns the **observation** — what the agent can "see"

**Response:**
```json
{
  "current_call_id": "CALL-E003-1",        ← The call you need to handle
  "transcript": "Building Fire. smoke at the hospital",  ← What the caller said
  "active_events": [],                       ← No events classified yet
  "resources": [                             ← Your available vehicles
    {"vehicle_id": "AMB-01", "type": "ambulance", "status": "available", ...},
    {"vehicle_id": "FIRE-01", "type": "fire_truck", "status": "available", ...},
    ...
  ],
  "step": 0,                                ← Step counter
  "calls_remaining": 14,                    ← How many calls are still coming
  "cases_resolved": 0,                      ← Score
  "reward": 0.0,                            ← Reward for this step
  "done": false                             ← Episode is not over
}
```

**Where did this data come from?**
- `transcript` → Generated by `call_generator.py` using random event types + noise
- `resources` → Created by `environment.py` based on task difficulty config
- `calls_remaining` → 15 calls were generated, 1 was loaded, 14 remain

---

### Step 2: Classify — Tell the System What This Call Is About

You read the transcript `"Building Fire. smoke at the hospital"` and decide this is CRITICAL.

```http
POST /step
Body: {"command": "CLASSIFY", "severity": "CRITICAL"}
```

**What happens inside the server:**
1. Compares your classification to the **hidden ground truth**
2. If correct → gives you +3.0 reward (triage signal)
3. Creates a new **event** in the city (status: "unserved")
4. Clears the current call, loads the **next** call from the queue
5. All waiting events accumulate waiting penalties
6. Returns the new observation

**Response:**
```json
{
  "current_call_id": "CALL-E001-2",         ← NEW call loaded automatically
  "transcript": "Car Accident. hurt at main street",
  "active_events": [
    {
      "event_id": "E003",                   ← The event YOU just created
      "severity": "CRITICAL",               ← The severity YOU assigned
      "location": "NODE_HOS",               ← Where it is (hidden from transcript)
      "resource_needed": "fire_truck",       ← What vehicle type is needed
      "steps_waiting": 1,                   ← It's been waiting 1 step
      "status": "unserved"                  ← Nobody dispatched yet!
    }
  ],
  "reward": 1.0,                            ← Your reward for this step
  "done": false
}
```

**Key insight:** After you classify, a new call is loaded. The event you created is now sitting in `active_events` waiting for a vehicle.

---

### Step 3: Dispatch — Send a Vehicle

Now you have an unserved event (E003, fire at the hospital). You should send a fire truck!

```http
POST /step
Body: {"command": "DISPATCH", "vehicle_id": "FIRE-01", "event_id": "E003"}
```

**What happens inside the server:**
1. Finds FIRE-01 and event E003
2. Checks: Is the vehicle available? ✅ Is the vehicle type correct? ✅ (fire_truck for fire)
3. Calculates shortest path through the city graph
4. Sets vehicle status to "en_route" with a path
5. Gives dispatch reward (+2.0 for correct vehicle type)
6. Returns new observation

**Response:**
```json
{
  "active_events": [
    {"event_id": "E003", "status": "unserved", "steps_waiting": 2}
  ],
  "resources": [
    {
      "vehicle_id": "FIRE-01",
      "status": "en_route",            ← Was "available", now driving
      "destination": "NODE_HOS",       ← Heading to the hospital
      "steps_to_arrival": 3,           ← Will arrive in 3 steps
      "path": ["NODE_A", "NODE_B", "NODE_HOS"]
    }
  ]
}
```

---

### Steps 4-6: Wait for Arrival

Each time you call `/step`, the vehicle moves one node along its path.

```http
POST /step
Body: {"command": "WAIT"}     ← or classify another call while waiting
```

After 3 steps, FIRE-01 arrives:
- Vehicle status changes to `"on_scene"`
- Environment checks: fire_truck at fire event? ✅
- Event status changes to `"resolved"`
- You get +30.0 reward (CRITICAL_RESOLVED from config)
- `cases_resolved` goes up by 1

---

### Repeat Until Done

Keep classifying calls and dispatching vehicles until:
- ✅ All calls processed AND all events resolved → **Success!**
- ❌ A CRITICAL event waits too long → **Critical failure, episode ends**
- ❌ Step count exceeds max_steps → **Timeout**

Then call `/grader` to get your final score.

## 4. All Endpoints Explained

| Endpoint | Method | What You Send | What It Does | What You Get Back |
| :--- | :--- | :--- | :--- | :--- |
| `/health` | GET | Nothing | Health check | `{"status": "healthy"}` |
| `/reset` | POST | `{"task_id": 1}` | Generates a new city + calls | First observation |
| `/step` | POST | An action (see below) | Simulates one step | New observation + reward |
| `/state` | GET | Nothing | Current episode metadata | Episode ID, step count, etc |
| `/tasks` | GET | Nothing | List available tasks | Task descriptions |
| `/grader` | POST | Nothing | Grades the finished episode | Score, fix rate, etc |
| `/baseline` | GET | Nothing | Runs a built-in bot on all tasks | Baseline scores |

### Actions You Can Send to `/step`

| Command | Required Fields | What It Does |
| :--- | :--- | :--- |
| `CLASSIFY` | `severity` | Classify the current call. Creates an event. |
| `DISPATCH` | `vehicle_id`, `event_id` | Send a vehicle to an event. |
| `MERGE` | `into_event_id` | Merge duplicate call into existing event. |
| `ESCALATE` | `event_id`, `severity` | Change an event's severity. |
| `DISCARD` | `call_id` | Throw away a non-emergency call. |
| `HOLD` | `event_id` | Put an event on hold (penalized for critical). |
| `RECALL` | `vehicle_id` | Recall a dispatched vehicle. |
| `REROUTE` | `vehicle_id`, `event_id` | Redirect a vehicle to a different event. |
| `WAIT` | Nothing | Do nothing this step (small penalty). |

## 5. The Reward System

The AI learns by maximizing reward. Here's how rewards work:

### You Get Points For:
| Action | Reward | Why |
| :--- | :--- | :--- |
| Correct severity classification | +3.0 | Good triage |
| Correct vehicle dispatched | +2.0 | Right tool for the job |
| CRITICAL event resolved | **+30.0** | Lives saved |
| SEMI_CRITICAL resolved | +15.0 | Injury treated |
| NORMAL resolved | +5.0 | Issue handled |

### You Lose Points For:
| Action | Penalty | Why |
| :--- | :--- | :--- |
| Underestimating severity | -5.0 | Dangerous mistake |
| Wrong vehicle type sent | -3.0 | Wasted resource |
| Each step a CRITICAL event waits | **-2.0/step** | People are dying |
| Each step a SEMI_CRITICAL waits | -1.0/step | People are suffering |
| CRITICAL event timeout (30+ steps) | **-20.0** | Total failure |
| Discarding a real emergency | -5.0 | People ignored |
| Doing nothing (WAIT) | -0.1 | Wasting time |

### Why Rewards Are Multi-Component
The reward has **5 independent signals** (not just one number):
```json
{
  "total": 28.0,
  "triage": 30.0,      ← Classification + resolution bonuses
  "dispatch": 2.0,     ← Vehicle matching
  "waiting": -4.0,     ← Accumulated waiting penalties
  "efficiency": -0.1,  ← Distance/time costs
  "penalties": 0.0     ← Rule violations
}
```
This prevents the AI from "gaming" a single reward signal.

## 6. Task Difficulty (Curriculum)

The 3 tasks have **different difficulty**, not just different names:

| | Task 1: Easy | Task 2: Medium | Task 3: Hard |
| :--- | :--- | :--- | :--- |
| **Events** | 5 | 10 | 15 |
| **Calls** | 15 | 40 | 60 |
| **Ambulances** | 4 | 3 | 2 |
| **Fire Trucks** | 3 | 2 | 1 |
| **Police** | 3 | 2 | 1 |
| **Total Vehicles** | 10 | 7 | 4 |
| **Critical Timeout** | 50 steps | 30 steps | 15 steps |
| **Max Steps** | 200 | 500 | 500 |

Task 1 is forgiving — many vehicles, few events, long timeout.
Task 3 is brutal — few vehicles, many events, short timeout.

## 7. How the Data Is Generated (Not Sent by You)

### The City (`city_graph.py`)
- 20 nodes (hospitals, intersections, stations, residential areas)
- Connected by edges with distances
- Dijkstra's algorithm calculates shortest paths
- Vehicles travel one edge per step

### The Calls (`call_generator.py`)
- Random event types: Building Fire, Car Accident, Medical Emergency, etc.
- Random locations on the city graph
- Random severities: CRITICAL, SEMI_CRITICAL, NORMAL
- **Noise added** to transcripts: fillers ("um", "oh god"), typos, truncation
- This noise is intentional — it tests whether the AI can extract the real info

### The Vehicles (`config.py` → `environment.py`)
- Placed at 3 stations around the city
- Each has a type (ambulance, fire_truck, police)
- Events need specific vehicle types (fire → fire_truck, accident → ambulance)

## 8. File Map — What Each File Does

```
Disaster_Management_RL_ENV/
│
├── config.py              ← All reward values + task difficulty settings
├── models.py              ← Pydantic schemas (request/response shapes)
├── city_graph.py           ← City map with nodes, edges, shortest paths
├── call_generator.py       ← Generates fake 911 calls with noise
├── reward.py               ← Multi-component reward calculator
├── client.py               ← Python client to call the API
├── collect_sft_data.py     ← Generates training data from baseline agent
├── openenv.yaml            ← OpenEnv hackathon manifest
├── Dockerfile              ← Container definition
├── Makefile                ← Build/run shortcuts
│
└── server/
    ├── __init__.py
    ├── app.py              ← FastAPI endpoints (/reset, /step, /grader, etc.)
    └── environment.py      ← THE CORE: all simulation logic lives here
```

### Data Flow:
```
You call /reset
    → app.py receives the request
    → calls environment.py reset()
        → city_graph.py builds the city
        → call_generator.py creates 911 calls
        → environment.py places vehicles
    → returns observation to you

You call /step with an action
    → app.py receives the request
    → calls environment.py step()
        → executes your action (classify/dispatch/etc)
        → moves vehicles along paths
        → checks if events are resolved
        → reward.py calculates reward breakdown
        → checks if episode is over
    → returns new observation + reward to you
```

## 9. Quick Start — Copy-Paste Examples

### Start the server:
```bash
uvicorn server.app:app --reload
```

### Play a full episode with curl:
```bash
# 1. Reset
curl -X POST http://localhost:8000/reset \
  -H "Content-Type: application/json" \
  -d '{"task_id": 1}'

# 2. Classify the first call
curl -X POST http://localhost:8000/step \
  -H "Content-Type: application/json" \
  -d '{"command": "CLASSIFY", "severity": "CRITICAL"}'

# 3. Dispatch a fire truck to the event
curl -X POST http://localhost:8000/step \
  -H "Content-Type: application/json" \
  -d '{"command": "DISPATCH", "vehicle_id": "FIRE-01", "event_id": "E003"}'

# 4. Wait for the vehicle to arrive
curl -X POST http://localhost:8000/step \
  -H "Content-Type: application/json" \
  -d '{"command": "WAIT"}'

# 5. Check baseline score
curl http://localhost:8000/baseline
```

### Play with Python:
```python
from client import EmergencyDispatchClient

with EmergencyDispatchClient("http://localhost:8000") as client:
    # Start episode
    obs = client.reset(task_id=1)
    print(f"First call: {obs['transcript']}")

    # Classify
    result = client.step(command="CLASSIFY", severity="CRITICAL")
    print(f"Reward: {result['reward']}")

    # Dispatch
    obs = result["observation"]
    event = obs["active_events"][0]
    result = client.step(
        command="DISPATCH",
        vehicle_id="FIRE-01",
        event_id=event["event_id"]
    )
    print(f"Vehicle dispatched, reward: {result['reward']}")
```

## 10. The Big Picture — Why This Exists

```
┌─────────────────────────────────────────────────┐
│              THE RL TRAINING LOOP                │
│                                                  │
│   ┌──────────┐     observation      ┌─────────┐ │
│   │          │ ──────────────────→  │         │ │
│   │ ENVIRON  │                      │  LLM    │ │
│   │ (this    │  ←─────────────────  │  AGENT  │ │
│   │ project) │      action          │         │ │
│   └──────────┘                      └─────────┘ │
│        │                                 ↑      │
│        │ reward                          │      │
│        └─────────── TRL TRAINER ─────────┘      │
│                  (updates weights)               │
└─────────────────────────────────────────────────┘
```

1. **This project** = the Environment (left box)
2. **Next step** = the LLM Agent (right box) — gets trained to make better decisions
3. **TRL + GRPO** = the Trainer — uses rewards to improve the LLM

The environment doesn't care WHO calls the API — a human with Postman, a curl script, or an LLM. It just simulates the world and returns observations + rewards.

**You are building the left box. The right box learns to play inside it.**
